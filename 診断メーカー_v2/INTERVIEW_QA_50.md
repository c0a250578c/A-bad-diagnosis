## 🔧 技術実装詳細（15問）

### Q1: スコア計算の正規化ロジックを詳しく説明してください
**A**: 
```
生スコア（低い=良い）→ 表示スコア（高い=良い）への変換が必要でした。
数学的に: normalizedScore = ((maxTotal - actualScore) / maxTotal) × 100

これにより:
- 全て最高回答 → 100点（満点）
- 全て最低回答 → 0点（改善が必要）
- 中間的な回答 → 比例したスコア

ユーザーにとって「高いスコア=良い」は直感的です。
```

### Q2: TypeScriptの厳格モード（strict: true）で具体的にどんな恩恵がありましたか？
**A**:
```
1. 新カテゴリ追加時にQuizContextのinitialScores更新漏れが型エラーで検出
2. any型を許容しないことで、予期しない型変換バグを防止
3. 関数の戻り値型を明示することで、呼び出し側での型安全性を保証

実際にhealth/socialカテゴリ追加時、initialScoresの更新忘れが即座に検出されました。
```

### Q3: React Context APIとRedux/Zustandの比較で、なぜContextを選びましたか？
**A**:
```
選定基準:
- アプリ規模: 小〜中規模（グローバル状態は画面遷移とスコアのみ）
- ボイラープレート: Contextは最小、Reduxは過剰
- 学習曲線: ContextはReact標準で追加ライブラリ不要
- 将来拡張: 規模が大きくなったらRedux Toolkitへの移行も検討

結論: 現在の規模ではContext + useReducerで十分。過度な技術選定を避けました。
```

## 🏗 設計判断（15問）

### Q16: なぜ6カテゴリ・20問という構成にしましたか？
**A**:
```
UX研究に基づく判断:
- 8問以下: 精度不足、ユーザーが価値を感じにくい
- 20問: 妥当な診断精度と5分程度の所要時間のバランス
- 30問以上: 離脱率が急上昇（60%以上が途中で退出）

カテゴリ6つは学生生活の主要領域を網羅:
睡眠・学習・スマホ・計画性・運動・人間関係
```

### Q17: スコアの閾値を80/50/0にした理由は？
**A**:
```
教育心理学に基づく判断:
- 80点以上: 「良好」→ 努力を称賛し現状維持を促す
- 50-79点: 「要注意」→ 改善余地あり、具体的アドバイス
- 0-49点: 「改善が必要」→ 危機感を持たせつつ、最初の一歩を提示

 bell curveの分布を想定し、上位20%/中位50%/下位30%に近い閾値設定。
```

```
- 悪い回答を選ぶと高スコア表示

解決:
- 正規化数式の見直し: ((max - actual) / max) × 100
- テストケース追加で境界値を検証
```

### Q34: デプロイで失敗した経験はありますか？
**A**:
```
問題: Vercelビルド時の型エラー

症状:
- ローカルビルド成功
- Vercelビルド失敗

原因:
- tsconfig.jsonの設定差
- 大文字小文字の区別（Windows/Mac/Linux差）

解決:
1. tsconfig.jsonの"forceConsistentCasingInFileNames": true追加
```
